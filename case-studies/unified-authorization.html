Pegasystems · Launchpad

Unified authorization

  • 2023
TL;DR Pega's authorization setup forced two completely different security models into separate, disconnected flows. I combined them into one. Validated as workable by security leadership and the architects who'd actually have to build it. Never got prioritized for implementation — that's the honest ending, not a launch story.

Context

Built for citizen developers — but not really

Meant for citizen developers configuring access control in Pega Launchpad. But realistically, setting up authorization requires someone who already understands the basics — role-based rules, attribute-based conditions, how the two interact. The product claimed to be approachable; the experience wasn't.

Problem

Flexibility built at the cost of friction

A heuristic review of the existing flow found several compounding problems. The system used Pega-specific terminology instead of language people already knew — so there was nothing to recognize, only things to learn. Settings had to be held in memory while switching between tabs, because nothing surfaced what had already been configured.

The interface wasn't built for anyone without deep familiarity — dense rows, horizontal scroll, no clear path through it. Even the documentation didn't help. And the two security models — role-based and attribute-based — lived in completely different places with no connection between them, even though any real authorization rule typically needs both.

Process

No direct user access — so go to the next best source

I didn't have direct access to the people who'd actually configure this at client companies. So I went to the next best source: solution architects, who build these systems for clients and see where people get stuck firsthand. They gave me real use cases and flagged where users would likely struggle.

Once I understood both models well enough to combine them, I ran a workshop with the Security stream director and the project owner — specifically to show that role-based and attribute-based rules could be merged into one model instead of staying two separate systems. From there, iteration went the same way as Fingerprinting: design reviews with the PO and architect, feedback folded back in. About 3 months total — roughly 1 month for heuristic analysis, research, architect calls, and design iteration specifically.

Solution

Two moves, not one

First: authorization configuration was pulled out of its own disconnected area and folded directly into the main flow of building a case type in Launchpad — no separate detour required.

Second: one view where role-based and attribute-based rules could be created together instead of separately. A rule like "tier 2 service specialists get elevated rights, and cases from a given location route to specialists in that location" becomes one configuration — not two systems to reconcile by hand.
